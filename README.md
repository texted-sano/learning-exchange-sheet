# 学習交流シート（プロトタイプ）セットアップガイド

コア機能＋最低限の本人確認・評価機能を実装した試作版です。「生徒が感想を入力・保存し、教員がコメント・10段階評価・いいねをつけ、生徒が後日見返せる」という一連の流れを、Googleスプレッドシートをデータの保存先として確認できます。

## 全体の構成

```
[index.html（フロントエンド）]  ──fetch(POST)──>  [Code.gs（Apps Script／バックエンドAPI）] ── 読み書き ──> [Googleスプレッドシート]
```

- **バックエンド（Code.gs）**：Googleスプレッドシートに紐づけたApps Scriptです。ウェブアプリとしてデプロイすると、外部から呼び出せるAPI（URL）になります。データの実体はスプレッドシートのシート（設定／名簿／授業／回答）です。
- **フロントエンド（index.html）**：単体のHTMLファイルです。将来「超教科書／未来スクールプラットフォーム」へ組み込むことを想定しているため、最初に「接続先URL（バックエンドのURL）」を入力して接続したあとは、その入力欄は画面から消え、「接続先を選び直す」ボタンに置き換わります。教師用ログインと生徒用ログインは試作確認をしやすくするため同じ画面にまとめてあります。

## セットアップ手順

### 1. スプレッドシートとバックエンドを用意する

1. 新しいGoogleスプレッドシートを作成します。
2. メニューの「拡張機能」→「Apps Script」を開きます。
3. 自動生成された `Code.gs` の内容をすべて削除し、本パッケージの `Code.gs` の内容を貼り付けて保存します。
4. スプレッドシートのタブに戻り、リロードします。しばらくすると「学習交流シート」というメニューが表示されます（出ない場合は数秒待つかもう一度リロード）。
5. 「学習交流シート」→「シートを初期化」を実行します。初回はGoogleから権限の確認を求められるので許可してください。これで「設定」「名簿」「授業」「回答」の4シートが作られます。

### 2. 設定と名簿を入力する

- 「設定」シート：クラス名、担任名、**教師パスワード**（先生としてログインする際に使います）を入力します。
- 「名簿」シート：B列に生徒の名前を1行1人で入力します（出席番号は任意）。**C列「パスワード」は空欄のままにしてください。** 生徒が初回ログイン時に自分でパスワードを設定します。

### 3. ウェブアプリとして公開する

1. Apps Scriptエディタ右上の「デプロイ」→「新しいデプロイ」を選びます。
2. 種類の選択で「ウェブアプリ」を選びます。
3. 「実行するユーザー」は **自分**、「アクセスできるユーザー」は **全員** に設定します（生徒・教員のどちらもGoogleログイン不要でアクセスできるようにするためです）。
4. デプロイすると、`https://script.google.com/macros/s/.../exec` という形のURLが発行されます。このURLが「Googleスプレッドシートへのアクセス情報」＝index.htmlに最初に入力する「接続先URL」です。

> **重要**：Code.gsの内容を後で修正したときは、保存するだけでは反映されません。「デプロイを管理」→ 編集（鉛筆アイコン）→ バージョン「新しいバージョン」を選んでデプロイし直してください。

### 4. フロントエンド（index.html）で確認する

1. `index.html` を開きます（ブラウザで直接開く、もしくはClaudeのプレビュー上で確認できます）。
2. 「接続先URL」に手順3で発行されたURLを貼り付けて「接続する」を押します。クラス名が表示されれば接続成功です（接続後はこの入力欄は消え、「接続先を選び直す」ボタンに切り替わります）。
3. **先生として確認する場合**：「先生としてログイン」タブでパスワードを入力 → 「授業を作成」でタイトルと発問を入れて保存 → 「授業一覧」から回答状況を確認し、10段階評価・いいね・コメントをつけて保存できます。「生徒管理」タブでは、スプレッドシートを開かずに、生徒の追加／名簿からの削除／パスワードのリセット（次回ログイン時に本人が再登録）／パスワードの直接設定（教師が決めて伝える）が行えます。
4. **生徒として確認する場合**：「生徒としてログイン」タブで名簿から名前を選ぶ →
   - パスワード未登録の生徒は「はじめてのログイン」フォームが表示されるので、4文字以上のパスワードを決めて登録します。
   - パスワード登録済みの生徒は、パスワードを入力してログインします。
   - 「授業」タブで公開中の授業に回答し、「きろくを見る」タブで自分の回答・先生の評価・いいね・コメントを見返せます。

## 今回追加した最低限の補強

- **生徒の本人確認をパスワード方式に変更**：名簿の「パスワード」列が空のときだけ、その名前で初回パスワード登録ができます。登録後は教師と同じ仕組み（名前＋パスワード）でログインします。
- **ログイン失敗のレート制限**：教師ログインはアプリ全体で、生徒ログイン・登録は名前ごとに、一定回数失敗すると数分間ロックします（総当たり攻撃対策）。CacheServiceで実装しているため、既存データには影響しません。
- **教師による生徒管理機能**：先生用ログインの「生徒管理」タブから、スプレッドシートを直接開かずに次の操作ができます。
  - 生徒の追加（出席番号・名前）
  - 名簿からの削除（過去の回答データ自体は「回答」シートに残ります）
  - パスワードのリセット（次回ログイン時に本人が再登録）
  - パスワードの直接設定（教師が決めたパスワードをその場で割り当てる）

  これは将来「スプレッドシートではなく出版会社のサーバーでデータを管理する」構成に移行することを見据えた設計でもあります。フロントエンドはすでに「アプリ内の操作だけで生徒情報を変更できる」前提で作られているため、バックエンドをスプレッドシートから別のサーバーAPIに置き換えても、フロントエンドの操作感は変えずに移行しやすくなっています。

### まだ残っている注意点（本運用前に検討してほしいこと）

- **パスワードは平文保存**：スプレッドシートのセルにそのまま保存されます。本運用ではハッシュ化や、超教科書プラットフォーム側の認証との連携を検討してください。
- **初回登録の「先着」問題**：名簿に名前があれば誰でも初回パスワードを登録できるため、本人より先に第三者が登録してしまう可能性があります。本運用では、教師が初期パスワードを配布する方式や、登録可能な期間を絞る運用ルールを検討してください。
- **レート制限はアプリ全体／名前単位の簡易版**：Apps Scriptから呼び出し元のIPアドレスを取得できないための簡略化です。

## 技術的な補足

- フロントエンドからバックエンドへの通信は、すべて `Content-Type: text/plain` でのPOSTにしています。これはApps Scriptのウェブアプリに対して、ブラウザの複雑なCORSプリフライト（OPTIONS）を発生させずに呼び出すための一般的な手法です。バックエンド側では `e.postData.contents` をJSONとして読み取って処理しています。
- 教師のセッションは持たせていません（ログイン後も、書き込み系のAPIを呼ぶたびにパスワードを一緒に送って毎回検証しています）。
- フロントエンドはブラウザのlocalStorage等を使わず、画面を開いている間だけメモリ上に状態を保持します。タブを閉じる／リロードすると接続情報やログイン状態は消えます。

## このあと拡張するとよい点（本実装に向けて）

- **認証の強化**：上記の「先着登録」問題の解消、ハッシュ化、超教科書プラットフォーム側のログインとの連携。
- **クラス／スプレッドシートの切り替え**：現状は1スプレッドシート＝1クラス想定です。複数クラス・複数年度を扱うなら、クラスコードでスプレッドシートを切り替える仕組みが必要です。
- **AIによる下書き生成**：Gemini APIなどを使い、教材文章から授業の発問・展開案や、コメントの下書きを自動生成する機能。
- **授業中のリアルタイム進行**：教員の操作に合わせて生徒の画面を一斉に切り替える、対話タイム専用モードなど。現在のシンプルな構成ではなく、一定間隔の自動更新やWebSocket的な仕組みの検討が必要です。
